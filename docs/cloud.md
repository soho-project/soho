##前言

微服务架构是个趋势；但是在实践中，微服务架构的实现还是有很多问题，比如：

- 服务之间通信
- 服务注册与发现
- 服务调用
- 服务监控
- 服务配置
- 服务追踪
- 服务网关
- 服务限流
- 服务容错
- 服务熔断
- 服务降级
- 服务网关

等等类似的问题。需要整改团队熟悉了解，统一行动，非常影响开发进度以及开发质量。
为了解决该类问题我重新定义了开发方式，业务开发按照传统方式分模块方式进行开发，唯一要求就是业务模块之间的调用一定要通过API接口方式调用。
剩下诸如上面的问题完全可以通过API桥接的方式解决。 如此一来在开发初期我们只需要单体项目的开发成本即可快速推进项目业务开发，后期采用极低的成本将单体服务进行拆分
